Angular: Ecosystems

The CLI

Interacting Solely w/ the keyboard and text, n o GUI

  • Most common approach is to use the
    BASH - (Bourne Again Shell)
  • Special installation instructions for getting BASH onto Windows machines
  • Usually you find yourself following a common pattern
    > [command] [options] [arguments]
    • Commands - the program you want to run
    • Options - change how the program works when you run the command
    • Arguments - the data you want to pass into the command when it runs
  • Usually there'll be a man page w/ usage instructions
    > man (command)

Operating Systems and Virtual Machines

Special software program that acts like a brain for your computer

  • It tells the hardware what to do, and how to act based on inputs and outputs, and the programs installed on your machine
  • Hardware - all the physical things you can touch and hold
  • Software - All the programs and instructions that tell your computer what to do
  • The OS sits between the hardware and the software, managing resources and giving the user a chance to interact

Virtual Machines

A software program that acts like a hardware interface for OS's

The big benefit is that they allow you and your team to install the exact same configuration across all your computers and servers, so the code you write will be written in the same environment.

  1. You instal the VM onto your computer
  2. Once installed your computer acts as the host machine, and each of the virtualOS is treated as a guest
  3. Guest OS's never touch the host - so you can configure each guest independently of your host machine

Aith Angular, the 2 most common Virtual Machines are:

  • VirtualBox - allows you to install and run full OS images, it's like installing a whole new operating system, but there are some downsides:
    • Very large files
    • Uses lots of memory
    • Slow to boot
    • Hard to share
  • Vagrant - solves the issues above by providing
    • A common platform and API to help share virtual machines
    • Vagrant Images - "Boxes"
      • Boot up fast (1 or 2 minutes)
      • Uses less hard drive space and memory than virtual machines - by allowing you to build a custom machine in layers, add the bits and pieces you need, and toss the rest
      • This makes virtual boxes much easier to share, update, and work with everyday.

Servers and Databases

They're at the center of every Angular app

  • Servers - are what sit on the backend of your Angular app, and usually wired into a network of some kind
  • Servers must be reliable
  • Go with a vendor that specializes in maintenance and upkeep of server hardware
  • Some good options are
    • Amazon Web Services (AWS)
    • Google Cloud Platform
    • Microsoft Azure
    • Digital Ocean
  • Basically, the internet is just a bunch of publically available servers that are accessed through a web browser
    • It sends requests and resources to the backend
    • Where the backend sends back data to the browser
    • This data is usually stored in a database:
      • They're like filing cabinets for your data
      • They make it easy to add and find entries based on keywords and search terms
      • There are 2 types of databases
        • SQL - Structured Query Language - Acts as a common interface for finding and filtering records in a database - great for large data sets and complex relationshhips because the data is stored in tables - examples - MySQL and PostgreSQL
        • NoSQL - Great for data sets with nested properties - or data that's being stored temporarily because the data can easily be passed back and forth between the front end and the back end - examples - CouchDB and Redis

Version Control

Aka: "Revision Control" or "Source Controll" are ways of managing changes in the code of your project across all the members of your team.

  • What was changed?
  • Who changed it?
  • How to undo it
  • How to unify the changes

Most Use Git

A distributed version control system

  • Very robust meaning everybody has a full copy of the history of all the changes to the code base
  • Making it very easy to restore is something happens to the original
  • Repo - an entire code base in version controll
  • Repos have branches - Independent versions of the code base
  • All branches are stored in the same repo
  • You make a new Independent copy of a Repo by cloning it, that may or may not stay insync w/ the original repo
  • Commits - are a set of changes across any # of files, these changes get recorded in the changelog of the repo
  • A conflict happens when there are 2 or more changes to the repo that can't be automatically figured out, and you have to manually fix and figure it out
  • You reconcile things by merging the different versions
  • Pull - when you download the latest version from a remote repo
  • Push - when you upload your version to a remote repo

Testing and Debugging

Angualr was built from the ground up for Testing and Debugging

Most common tests

  • Code linting - checking code for readability, consistency, and syntax
    The never run your code, they just read over every line in a file and checks them against rules
  • Unit testing - checking small chunks of code to make sure they work as expected, they make it clear as to what your code is meant to do, and lowers the risk of new code causing errors, they are usually:
    • Functions
    • Objects
    • Methods
    Angular uses Jasmine framework w/ the Karma task runner
  • End-to-end testing - "Integration tests" - used for testing user interaction - makes sure the UI is meant to work the way it's supposed to by pretending to be an end user
    • Angular uses the Protractor Framework for end-to-end testing

Debuggin

Almost always happens in the browser, in some sort of dev tool that let's you see reports of the errors

Angular does a great job of telling you what's wrong

Continuous Integration, Delivery and Deployment

Involved in a workflow that reduces risk and improve code quality

  • CI - Process of committing small chunks of code to a shared repository several times a day
    • Each commit is tested for bugs using the automated testing process
    • If there's a bug it's easy to rewind to a functional state
    Common Platforms are: (they all build and test code in an automated way) - Automation gives you confidence!
    • Travis CI
    • Codeship
    • Jenkins
    • Circle CI
  • CD(elivery) - the confidence to deliver, keeping your code in a state of readyness that can be pushed to production at any time - while you still get to choose when you push code into production
  • CD(eployment) - automates the deployment process - every new change, no matter how small is pushed right to production as soon as it passes the test suite - while it takes a way some control, it also removes a step from the build deploy process so you can focus on building new features for your app
